# v0.17 Slice C — Scene Registry & Constructor Navigation — Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Replace instance-based scene navigation (`app.start(new GameScene())`, `scenes.setScene(sceneInstance)`) with constructor-based, typed navigation (`app.start(GameScene, data?)`, `scenes.setScene(GameScene, data?, options?)`) backed by a validated `ApplicationOptions.scenes` registry, per `.workspace/specs/exojs-v0.17-final-core-spec/01-implementation-spec.md` §6, §10.1–10.2, §11.1, Slice C (§16).

**Architecture:** A new `SceneTypes.ts` owns the constructor/inference typing (`SceneConstructor<Data>`, `InferSceneData`, `SetSceneArgs`) and a runtime `resolveSetSceneArgs()` helper that disambiguates the variadic `(data?, options?)` tail at runtime (TypeScript erases `Data`, so this is a real runtime concern, not just typing — see Task 1). `SceneDirector` gains an internal `Map<AnySceneConstructor, string>` registry, validated once at construction from `ApplicationOptions.scenes`; `setScene`/`Application.start` become generic over the target constructor. `SceneScope<Data>` already threads `Data` through `prepare(data)` (built in Slice B) — this slice is the first to actually supply a non-`void` `Data` value. The `currentScene` setter is deleted; `Application.stop()`/`destroy()` move to a new internal `SceneDirector._clearScene()`.

This slice's clean break (`app.start(new X())` no longer compiles) requires updating every call site across `examples/`, `site/src/content/guides/`, and `packages/create-exo-app/` in the same PR — CI's `typecheck:examples`/`typecheck:guides` gate the merge, and slices land independently-green (spec §16 dependency graph). That mechanical sweep is Task 8, run as a script, not hand-edited per file.

**Tech Stack:** TypeScript (strict), Vitest (`expectTypeOf` for type tests), existing `SceneDirector`/`SceneScope`/`Scene` classes from Slices A/B (PRs #390–#392).

## Global Constraints

- Clean breaks only — no deprecated aliases, no shims (pre-1.0 policy, spec §1). `Scene` instances are never passed to `setScene`/`start` again.
- `Data` is fully erased at runtime (`declare private readonly _sceneData?: Data` on `Scene<Data>`, Slice B). No behavior may depend on runtime knowledge of whether `Data` is `void`.
- Registry validation and unregistered-target rejection are **dev-build only** (`__DEV__`), matching existing `assert()`/`invariant()` conventions in `src/core/dev.ts`. Registration checks must NOT construct instances (spec §6.3 — side effects).
- No `Routes` generic anywhere; `Application`/`SceneDirector` stay non-generic (spec §6.2).
- New public error classes are named, extend `Error`, follow the `RenderError` shape (`src/rendering/RenderError.ts`): `public readonly name` set in constructor, documented fields, exported from `src/core/index.ts`.
- JSDoc conventions: see `[[feedback-jsdoc-conventions]]` memory — every public export gets a doc comment; `@internal` for engine-only surface.
- `pnpm docs:api:generate` must be run and committed before the final push (push-gated).
- Every task ends green on its own scoped test command before moving to the next.

This is the trickiest task in the slice: **runtime disambiguation of the variadic tail.** `SetSceneArgs<Data>` is `[options?] | [data, options?]` depending on whether `Data extends void` — but that distinction is compile-time-only. At runtime, `setScene(GameScene, someObject)` and `setScene(TitleScene, someObject)` both just receive `args = [someObject]`, with no runtime signal for which case applies. The resolution: `SetSceneOptions` is a small closed-shape object (`transition?`, `retainCurrent?`); we treat `args[0]` as _options_ only if it structurally has no keys outside that set, otherwise as _data_. Document the resulting edge case (a `Data` shape whose only keys happen to be named `transition`/`retainCurrent` is misread as options) directly on `SetSceneArgs`'s JSDoc — this is an accepted, documented tradeoff, not a bug to "fix" here.

```ts
// src/core/SceneTypes.ts
import type { Color } from './Color';
import type { Scene } from './Scene';

/**
 * Zero-argument constructor for a {@link Scene} subclass. `Data` is the
 * activation-data type this scene expects, inferred at navigation call
 * sites via {@link InferSceneData}.
 */
export interface SceneConstructor<Data = void> {
  new (): Scene<Data>;
}

/** A {@link SceneConstructor} for any activation-data type — used for registry storage and navigation targets whose `Data` is not statically known. */
// eslint-disable-next-line @typescript-eslint/no-explicit-any -- deliberately erases Data for heterogeneous registry storage; see spec §6.3.
export type AnySceneConstructor = SceneConstructor<any>;

/** Extracts the activation-data type a {@link SceneConstructor} expects. */
export type InferSceneData<C> = C extends SceneConstructor<infer Data> ? Data : never;

/**
 * Fade-to-color scene transition. The screen fades to `color` (default black)
 * over `duration` ms (default 220), the scene change happens at full
 * opacity, then the screen fades back in.
 */
export interface FadeSceneTransition {
  type: 'fade';
  duration?: number;
  color?: Color;
}

/** Discriminated union of supported {@link SceneDirector} transitions. */
export type SceneTransition = FadeSceneTransition;

/** Options passed to {@link SceneDirector.setScene} / {@link Application.start}. */
export interface SetSceneOptions {
  transition?: SceneTransition;
}

/**
 * The reserved key set of {@link SetSceneOptions}. Used at runtime by
 * {@link resolveSetSceneArgs} to distinguish an options argument from a data
 * argument when only one tail argument is present (`Data` is erased at
 * runtime — see that function's doc for the full explanation).
 */
const setSceneOptionsKeys = new Set(['transition', 'retainCurrent']);

/**
 * Tuple type for the variadic tail of `setScene`/`start` calls, after the
 * target constructor. When `Data` is `void` there is no data slot — only
 * `options`. Otherwise `data` is required and `options` is optional.
 *
 * **Runtime caveat:** `Data` has no runtime representation (erased). A
 * single tail argument is classified as *options* when it is a plain object
 * whose keys are a subset of `{ transition, retainCurrent }` (see
 * {@link resolveSetSceneArgs}) — otherwise it is classified as *data*. A
 * scene whose `Data` shape's only keys happen to be named `transition`
 * and/or `retainCurrent` would be misread as options; avoid that shape, or
 * nest such fields one level deeper.
 */
export type SetSceneArgs<Data> = [Data] extends [void] ? [options?: SetSceneOptions] : [data: Readonly<Data>, options?: SetSceneOptions];

const isPlainObject = (value: unknown): value is Record<string, unknown> => typeof value === 'object' && value !== null && !Array.isArray(value);

const looksLikeSetSceneOptions = (value: unknown): value is SetSceneOptions =>
  isPlainObject(value) && Object.keys(value).every(key => setSceneOptionsKeys.has(key));

/**
 * Resolve the erased-at-runtime `(data?, options?)` tail of a `setScene`/
 * `start` call into its two parts. See {@link SetSceneArgs} for the
 * disambiguation rule and its documented edge case.
 * @internal
 */
export function resolveSetSceneArgs(args: readonly unknown[]): { data: unknown; options: SetSceneOptions } {
  if (args.length >= 2) {
    return { data: args[0], options: (args[1] as SetSceneOptions | undefined) ?? {} };
  }

  if (args.length === 1) {
    return looksLikeSetSceneOptions(args[0]) ? { data: undefined, options: args[0] } : { data: args[0], options: {} };
  }

  return { data: undefined, options: {} };
}

/**
 * Thrown (dev builds only) when `ApplicationOptions.scenes` registers the
 * same constructor under more than one key.
 */
export class DuplicateSceneRegistrationError extends Error {
  public readonly constructorName: string;
  public readonly keys: readonly [string, string];

  public constructor(constructorName: string, keys: readonly [string, string]) {
    super(
      `Scene constructor "${constructorName}" is registered under more than one key in ApplicationOptions.scenes: "${keys[0]}" and "${keys[1]}". Each scene constructor may be registered only once.`,
    );
    this.name = 'DuplicateSceneRegistrationError';
    this.constructorName = constructorName;
    this.keys = keys;
  }
}

/**
 * Thrown (dev builds only) when `ApplicationOptions.scenes` contains a value
 * that is not a {@link Scene} subclass constructor.
 */
export class InvalidSceneRegistrationError extends Error {
  public readonly key: string;

  public constructor(key: string) {
    super(`ApplicationOptions.scenes["${key}"] must be a Scene subclass constructor.`);
    this.name = 'InvalidSceneRegistrationError';
    this.key = key;
  }
}

/**
 * Thrown (dev builds only) when navigating to a constructor that is not
 * present in `ApplicationOptions.scenes`.
 */
export class UnregisteredSceneError extends Error {
  public readonly constructorName: string;
  public readonly registeredNames: readonly string[];

  public constructor(constructorName: string, registeredNames: readonly string[]) {
    const list = registeredNames.length > 0 ? registeredNames.join(', ') : '(none)';

    super(`Scene constructor "${constructorName}" is not registered in ApplicationOptions.scenes. Registered scenes: ${list}.`);
    this.name = 'UnregisteredSceneError';
    this.constructorName = constructorName;
    this.registeredNames = registeredNames;
  }
}

/**
 * Validate and index an `ApplicationOptions.scenes` record: every value must
 * be a function whose prototype chain includes {@link Scene} (checked via
 * `prototype instanceof Scene` — deliberately never constructs an instance,
 * since construction may have user side effects), and no constructor may
 * appear under more than one key. Dev builds only; production builds skip
 * validation and return an empty map when `scenes` is undefined, or an
 * unvalidated map otherwise (the map is still needed at runtime for key
 * diagnostics-free navigation).
 * @internal
 */
export function validateSceneRegistry(
  scenes: Record<string, AnySceneConstructor> | undefined,
  SceneBase: typeof Scene,
): ReadonlyMap<AnySceneConstructor, string> {
  const registry = new Map<AnySceneConstructor, string>();

  if (scenes === undefined) {
    return registry;
  }

  for (const [key, ctor] of Object.entries(scenes)) {
    if (__DEV__ && !(typeof ctor === 'function' && ctor.prototype instanceof SceneBase)) {
      throw new InvalidSceneRegistrationError(key);
    }

    const existingKey = registry.get(ctor);

    if (__DEV__ && existingKey !== undefined) {
      throw new DuplicateSceneRegistrationError(ctor.name, [existingKey, key]);
    }

    registry.set(ctor, key);
  }

  return registry;
}
```

### 3.4 — `setScene`: typed, constructor-based

Replace the entire `setScene` method with:

```ts
  /**
   * Switch to a fresh instance of `target`, ending the previously active
   * scene permanently. No-op when `target` is already the active scene's
   * constructor *and* no data/options are meaningfully different — actually,
   * per definition §11.4, ordinary switching always creates a fresh
   * instance, so same-constructor calls still swap in a new instance (this
   * differs from the old instance-identity no-op check, which is gone along
   * with instance-based navigation). An optional fade transition runs the
   * swap at full opacity. The new scene completes `load()`+`init()` before
   * the old one is torn down, so there is no blank frame between them, and
   * the outgoing scene keeps running until the switch boundary is crossed.
   *
   * Rejects (dev builds) with {@link UnregisteredSceneError} when `target`
   * is not present in `ApplicationOptions.scenes`.
   */
  public async setScene<C extends AnySceneConstructor>(target: C, ...args: SetSceneArgs<InferSceneData<C>>): Promise<this> {
    if (__DEV__ && !this._registry.has(target)) {
      throw new UnregisteredSceneError(target.name, [...this._registry.values()]);
    }

    const { data, options } = resolveSetSceneArgs(args);

    await this._runWithTransition(async () => {
      const scene = new target();
      const newScope = await this._prepareScene(scene, data);
      const previousScope = this._activeScope;

      this._activeScope = newScope;

      if (previousScope !== null) {
        await this._disposeScene(previousScope);
      }

      newScope.activate();

      this.onChangeScene.dispatch(scene);
      this.onStartScene.dispatch(scene);
    }, options.transition);

    return this;
  }

  /**
   * @internal Clear the active scene (if any) without activating a new one.
   * Replaces the old `setScene(null)` path — used only by
   * {@link Application.stop} / {@link Application.destroy}, never part of
   * the public navigation surface (definition §10.1: navigation always
   * targets a registered constructor).
   */
  public async _clearScene(): Promise<this> {
    await this._runWithTransition(async () => {
      const previousScope = this._activeScope;

      this._activeScope = null;

      if (previousScope !== null) {
        await this._disposeScene(previousScope);
      }

      this.onChangeScene.dispatch(null);
    });

    return this;
  }
```

Note the removed same-scene no-op check: the old code compared `scene === (this._activeScope?.scene ?? null)` to skip redundant switches. That check is inherently tied to instance identity, which the new API doesn't expose to callers (they pass a constructor, never see the instance) — definition §11.4 confirms "ordinary switching always creates a fresh instance," so removing the no-op is the spec-conformant behavior, not an oversight.

```js
#!/usr/bin/env node
// Throwaway codemod for v0.17 Slice C: rewrites the mechanical
// `app.start(new XScene())` → registry + `app.start(XScene)` pattern across
// examples/site-guides/create-exo-app. Delete this file once Task 8 lands —
// it is not part of the shipped repo tooling.
import { readFileSync, writeFileSync } from 'node:fs';
import { globSync } from 'node:fs';
// Node 22+ has fs.globSync; if the pinned Node version predates it, swap for
// `glob` (already a transitive devDependency — check `pnpm why glob` first).

const startCallPattern = /app\.start\(new (\w+)\(\)\)/;
const newApplicationPattern = /new Application\(\{/;

const targets = [...globSync('examples/**/*.{ts,js}'), ...globSync('site/src/content/guides/**/*.{ts,md,mdx}'), ...globSync('packages/create-exo-app/**/*.ts')];

let changed = 0;
let skipped = [];

for (const file of targets) {
  const source = readFileSync(file, 'utf8');
  const startMatch = startCallPattern.exec(source);

  if (startMatch === null) {
    continue; // no app.start(new X()) in this file — leave untouched
  }

  const sceneName = startMatch[1];
  const appMatch = newApplicationPattern.exec(source);

  if (appMatch === null) {
    skipped.push(`${file}: found "${startMatch[0]}" but no "new Application({" to inject scenes into`);
    continue;
  }

  // Inject `scenes: { X },` as the first property of the Application options
  // object literal (shorthand — key name equals the imported class name).
  const withRegistry =
    source.slice(0, appMatch.index + appMatch[0].length) + `\n    scenes: { ${sceneName} },` + source.slice(appMatch.index + appMatch[0].length);

  const withStartCall = withRegistry.replace(startCallPattern, `app.start(${sceneName})`);

  writeFileSync(file, withStartCall);
  changed++;
}

console.log(`Rewrote ${changed} files.`);

if (skipped.length > 0) {
  console.log(`\nSkipped (needs manual handling):`);
  for (const line of skipped) console.log(`  ${line}`);
}
```
